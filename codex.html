Codex 基礎教學

給初階工程師的實作導向介紹：理解 Codex、下對指令、完成文件整理與程式碼開發任務。

Learning Goals

今天結束後，你要能做到三件事

  • 知道 Codex 和一般聊天 AI 的差異，以及它適合處理哪些工程任務。
  • 會寫清楚的指令，讓 Codex 能讀檔、改檔、跑測試、整理結果。
  • 能用 Codex 完成一個小型文件整理任務與一個小型程式碼任務。
建議時間：3 分鐘
What Is Codex

Codex 是能操作專案的 AI 工程協作者

Read

讀懂專案

瀏覽資料夾、閱讀檔案、找出入口點、理解程式結構。

Change

修改內容

編輯程式碼、文件、設定檔，並盡量遵守現有風格。

Verify

驗證結果

執行指令、跑測試、看錯誤訊息，回報修改結果。

建議時間：4 分鐘
Mental Model

把 Codex 當成會問問題、會動手的 Junior Pair

  • 你提供目標、背景、限制與驗收標準。
  • Codex 負責探索、提出修改、執行命令與回報結果。
  • 你仍然要審查重要變更，尤其是刪除資料、改設定、部署與安全相關操作。
不要只說：「幫我修好」
改成：「這個 API 在空資料時會 500，請找出原因並修正，最後跑測試確認。」
建議時間：4 分鐘
Workspace

使用前先理解工作區

Codex 看得到什麼

  • 目前專案資料夾中的檔案
  • 終端機指令輸出
  • 工具回傳的檢查結果

你要注意什麼

  • 確認目前資料夾是正確專案
  • 重要修改前先有 git 狀態
  • 不要把 secret 寫進 prompt 或檔案
常用起手式：
「先幫我看這個專案結構，不要修改檔案，整理出入口檔案與主要模組。」
建議時間：5 分鐘
Command Types

初學者最常用的 6 類指令

理解

Explain / summarize / trace flow

修改

Implement / update / refactor

除錯

Debug / fix error / inspect logs

測試

Run tests / add tests / verify

文件

README / guide / changelog

版本控制

Diff / commit message / PR summary

建議時間：5 分鐘
Slash Commands

像 /model 這類指令，是控制 Codex 工作階段

/help查看目前版本支援的指令。
/model切換模型與推理強度。
/status查看模型、權限與用量。
/permissions調整 Codex 可以做哪些操作。
/init建立 AGENTS.md 專案規則。
/review針對目前變更做 code review。
/compact對話很長時壓縮上下文。
/new開啟新的想法或新任務。

教學提醒：Slash command 是「控制介面」，自然語言 Prompt 是「交代任務」。

建議時間：5 分鐘
Explain Prompts

理解專案：先讓 Codex 建立地圖

請先閱讀這個專案，不要修改檔案。

請回覆：
1. 專案用途
2. 主要資料夾與檔案
3. 程式入口點
4. 本機如何啟動
5. 你建議我先看哪 3 個檔案
  • 適合接手新專案、讀別人的程式、快速找方向。
  • 先要求「不要修改檔案」，可以把探索和實作分開。
建議時間：4 分鐘
Edit Prompts

修改功能：給目標，也給限制

請幫我在會員列表加入搜尋功能。

限制：
- 不要改資料庫 schema
- 優先沿用現有 component style
- 搜尋欄位包含 name 與 email

完成後：
- 說明改了哪些檔案
- 跑現有測試或提供無法跑的原因
  • 好的限制能減少過度改動。
  • 要求驗證，讓任務不只停在「看起來有改」。
建議時間：5 分鐘
Debug Prompts

除錯：提供錯誤、重現步驟、期望行為

現在執行 `npm test` 會失敗。

請你：
1. 先跑測試確認錯誤
2. 找出最可能的原因
3. 做最小修改修正
4. 再跑一次測試
5. 用條列說明 root cause
不清楚

「壞了，幫我看。」

清楚

「這個 command 失敗，請重現、定位、最小修正、驗證。」

建議時間：5 分鐘
Verify Prompts

驗證不是附加工作，是工程工作的一部分

  • 請 Codex 跑既有測試，例如：
  • 沒有測試時，要求它提供手動驗收步驟。
  • 前端任務可要求截圖檢查、RWD 檢查、互動流程檢查。
npm test pytest go test ./... 
請完成修改後驗證：
- 能跑自動測試就跑
- 無法跑請說明原因
- 補上 3 個我可以手動檢查的步驟
建議時間：4 分鐘
Prompt Formula

好指令的 5 個欄位

1. 目標要完成什麼
2. 背景目前狀況與相關檔案
3. 限制不能改什麼、偏好什麼
4. 輸出要程式、摘要、表格或步驟
5. 驗收如何確認完成
模板：
請幫我 [目標]。
背景是 [目前狀況]。
限制是 [限制條件]。
完成後請 [輸出格式]。
驗收方式是 [測試或檢查方式]。
建議時間：5 分鐘
Demo A

現場示範：文件與資料夾整理

Step 1請 Codex 掃描資料夾
Step 2分類檔案與命名問題
Step 3產生整理方案
Step 4建立 README 或索引
Step 5檢查結果
示範指令：
請掃描目前資料夾，不要移動檔案。
幫我整理出檔案類型、可能用途，以及建議的分類資料夾。
最後產生一份 README 草稿，說明這個資料夾內容。
建議時間：7 分鐘
Demo B

現場示範：程式碼專案開發

Step 1理解架構
Step 2指定小功能
Step 3修改程式碼
Step 4執行測試
Step 5看 diff 與總結
示範指令：
請先看這個專案架構，找出前端入口與主要元件。
接著幫我新增一個搜尋欄位，能用關鍵字過濾列表。
請沿用現有樣式，完成後跑測試或提供手動驗收步驟。
建議時間：8 分鐘
Practice

學員練習：兩選一，15 分鐘內完成

文件任務

整理課程資料夾

請 Codex 掃描資料夾、分類內容、產生 README，並列出後續整理建議。

程式任務

新增一個小功能

請 Codex 理解專案後，新增搜尋、排序、空狀態或錯誤提示其中一項。

練習要求：
- 先讓 Codex 解釋，不要直接改
- 再給明確限制與驗收方式
- 完成後請它總結修改與測試結果
建議時間：10 分鐘
Review

看 Codex 的成果時，先看這些

  • Scope：是否只改了任務需要的範圍？
  • Behavior：是否真的符合原本目標？
  • Tests：是否有跑測試，或至少有手動驗收步驟？
  • Risk：是否改到設定、刪資料、改 API contract 或安全邏輯？
  • Diff：重要任務一定要看變更差異。
建議時間：4 分鐘
Common Mistakes

初學者常見錯誤

避免
  • 一次丟太大任務
  • 沒有說限制
  • 沒有要求驗證
  • 不看 diff 就接受修改
改成
  • 拆成探索、修改、驗證
  • 列出不能碰的範圍
  • 要求測試與手動檢查
  • 請它解釋修改理由
建議時間：4 分鐘
Best Practices

跟 Codex 協作的基本節奏

Explore先讀懂，不急著改
Plan確認修改範圍
Implement小步修改
Verify測試與手動檢查
Summarize整理成果與風險
推薦工作流：
「先分析，不修改」→「提出方案」→「實作」→「跑測試」→「總結 diff」
建議時間：4 分鐘
Takeaway Template

課後可直接複製的 Prompt

請先閱讀目前專案，不要修改檔案。
請說明專案架構、入口點、啟動方式，以及你建議先看的檔案。

接著請幫我完成：[具體任務]
限制：
- [限制 1]
- [限制 2]

完成後請：
1. 說明改了哪些檔案
2. 跑可用的測試
3. 提供手動驗收步驟
4. 列出你認為還有風險的地方
建議時間：2 分鐘
Closing

把 Codex 用好，不是靠一句神奇指令

  • 清楚描述目標，讓它知道終點。
  • 給出限制，讓它知道邊界。
  • 要求驗證，讓結果能被檢查。
  • 審查 diff，保留工程師的判斷權。

最實用的心法：把大任務拆成「理解、修改、驗證、總結」。

建議時間：2 分鐘
