## 核心觀念：傳統程式 vs 生成式 AI

要跟工程師解釋生成式 AI，最快的方法就是對比他們每天在寫的 Code：

### 1. 傳統程式（Traditional Programming）：規則驅動

在過去，我們寫程式是 **「輸入資料 + 寫死規則（If/Else、演算法）= 輸出結果」**。

* **邏輯：** 工程師扮演上帝。你必須把所有的邏輯（Logic）、邊界條件（Edge Cases）全部用程式碼定義好。如果漏寫了一個 if，系統就會噴 Bug。

### 2. 生成式 AI（Generative AI）：機率與預測驅動

生成式 AI 顛覆了這個過程。它是 **「餵入海量資料 + 模型訓練 = 機器自己找出潛在規則」**。當我們給它一個輸入（Prompt）時，它是基於這些規則，去「憑空創造（Generate）」出世界上原本不存在的新內容。

* **邏輯：** 工程師不再寫死規則，而是給予「提示與上下文」。AI 本質上是一個「超級文字接龍機」**或**「像素預測器」，它會根據你給的前文，去計算下一個字、下一個像素「最有可能出現的是什麼」，進而組合出一段程式碼、一張圖或一篇文章。

---

> 簡而言之：傳統 AI 是拿來做**分類與判斷**的（例如：判斷這封郵件是不是垃圾信、這張照片是不是貓）；而生成式 AI 是拿來**創造新東西**的（例如：幫我寫一封感謝信、畫一隻在太空的貓）。

## 工程師最需要知道的生成式 AI 三大核心底層

在課程的開頭，您可以幫工程師科普這三個核心名詞，他們聽完一定會心領神會：

### 1. 什麼是 LLM（大型語言模型）？

它是生成式 AI 的大腦（如 GPT-4, Claude 3.5）。你可以把它想像成一個「讀完了整個網際網路、壓縮過的超大型資料庫」。

* **對工程師的隱喻：** 它就像一個熟讀了 Stack Overflow、GitHub 所有開源專案、以及無數技術文件的「超級實習生」。它雖然什麼都看過，但他沒有獨立思考能力，他只是在「模仿」人類寫出正確答案機率最高的文字。

### 2. 什麼是 Token（標記）？

這是生成式 AI 計算成本與理解文字的最小單位。AI 其實看不懂中文或英文，它必須把文字切成一塊一塊的 Token，再轉成電腦懂的數字（向量）。

* **對工程師的隱喻：** 這就是為什麼 AI 計費是用 Token 計算，而不是用字數。而且這也解釋了為什麼 AI 有「Context Window（上下文視窗）」的限制——因為它的記憶體快取（Cache）是有限的，超過一定數量的 Token 它就會開始忘記前面的對話。

### 3. 什麼是「提示詞工程（Prompt Engineering）」？

對工程師來說，Prompt 其實就是一種「新型態的程式語言」。

* **對工程師的隱喻：** 過去我們用 Python 或 JavaScript 對機器下指令；現在我們用「人類的自然語言」對 AI 下指令。Prompt Engineering 的本質，就是透過精確的變數設定（Role, Context, Constraints），來**收斂 AI 的機率分布**，讓它不要瞎掰（幻覺），乖乖輸出我們要的正確結果。

## 詳細內容


### 什麼是 TOKEN


在大型語言模型中，**Token 是模型處理與運算文字的最基本單位**。當我們向模型輸入一段句子時，這段文字會先經過一個稱為 Tokenization（分詞）的過程，被切分成一個個的 Token。

關於 Token 的特性與在模型中的運作方式，包含以下幾個重要概念：

*   **與字詞的對應關係不固定：** Token 並不完全等同於人類認知中的一個「字」或一個「詞」。以中文為例，很多時候好幾個中文字合起來才會構成一個 Token，甚至有時候單一箇中文字反而會被切分成好幾塊 Token。以英文而言，字母的大小寫或前面是否帶有空格，都會被模型視為完全不同的 Token，例如「Apple」、「APPLE」以及帶有前綴空格的「 Apple」都是各自獨立的 Token。此外，標點符號如驚嘆號、雙引號等，本身也是獨立的 Token。
*   **對應唯一的數字編號（ID）：** 切分出來的每一個 Token 都會對應到一個專屬的整數編號（ID）。例如在 Llama 模型內部，編號 0 的 Token 代表的就是驚嘆號。
*   **轉換為高維度向量（Token Embedding）：** 語言模型無法直接運算文字或單純的 ID 數字，因此會透過一個稱為 Embedding Table 的龐大矩陣進行「查表」，**將每一個 Token 的 ID 轉換成一排數字，也就是一個高維度的向量（Vector）**。
*   **蘊含語意相似性：** 這些 Token 轉換成的向量並非隨機給予，**意義相近的 Token 會獲得在空間中距離相近的數學向量**。舉例來說，當模型看到「Apple」這個 Token 並將其轉換為向量後，如果去計算相似度，會發現它跟「MacBook」、「iPhone」或是中文「蘋果」的 Token 向量非常接近。
*   **語言模型的最終目標：** 語言模型運作的核心任務，就是讀取前面已經輸入的一連串 Token，經過層層的神經網路計算後，**去預測接下來的句子應該接上字典裡哪一個 Token 的機率最高**。


#### 可以用平易的觀念或是範例 解釋給我知道什麼是 Token 嗎？
我們可以把 Token 想像成是語言模型在閱讀和寫作時使用的**「文字積木」**。模型其實看不懂我們人類的完整句子，它必須先把句子拆解成一塊一塊的積木（Token），才能開始進行數學運算。

以下用幾個平易近人的範例來幫你理解 Token 的運作方式：

*   **積木是怎麼切的（不一定等於一個字）：** 
    對模型來說，積木的切法跟我們人類直覺有時不太一樣。以中文來說，雖然為了方便理解，我們常假設一個中文字就是一個 Token，但實際上，有時候好幾個中文字合起來才是一塊 Token，也有時候一個中文字會被切碎成好幾塊 Token。
    以英文來說，大小寫或是有沒有空格，都會被視為形狀完全不同的積木。例如首字母大寫的「Apple」、全部大寫的「APPLE」，以及前面多帶一個空白鍵的「 Apple」，對模型來說都是三塊各自獨立的 Token。就連標點符號，例如「驚嘆號（！）」本身也是一塊獨立的 Token。

*   **積木都有專屬編號（Token ID）：** 
    因為電腦只懂數字，所以切出來的每一塊文字積木，都會被貼上一個專屬的數字標籤（ID）。例如在知名的 Llama 模型中，編號 0 號的天字第一號 Token，其實就是「驚嘆號」。

*   **積木會互相連結（蘊含相關意義）：** 
    這些積木並不是死板板的符號，模型會把它們轉換成帶有意義的空間位置。舉例來說，當模型拿到「Apple」這塊積木時，它會發現這塊積木跟代表水果的「banana（香蕉）」靠得很近。但同時，模型也知道蘋果公司的存在，所以「Apple」這塊積木跟「MacBook」、「iPhone」、中文的「蘋果」，甚至是蘋果總部所在地「Cupertino」都非常接近。
    中文也是如此，假如你給模型一個代表姓氏「王」的 Token，模型算出來的結果會顯示它知道「王」對應到英文的「King」，同時也跟另一個中文姓氏「黃」的意思很接近。

*   **語言模型的工作就是「猜下一塊積木」：** 
    你可以把語言模型想像成一個超強的「文字接龍」機器。當你輸入一段話，模型會把這些積木綜合起來看。例如當你輸入「兩顆青蘋」，模型會考量前面這幾塊積木（兩、顆、青、蘋）的意思，並推算接下來接上「果」這塊積木的機率最高。再舉個例子，如果輸入「天氣」，模型在內部經過層層思考後，可能會預測下一個積木應該接「預」，再下一個積木接「報」，最後產出「天氣預報」這個詞。

簡單來說，**Token 就是語言模型用來拆解、理解和預測人類語言的最基礎文字積木**。


### LLM（大型語言模型）是什麼？它是怎麼運作的？

大型語言模型（LLM）本質上就是一個複雜的數學函式，**它的核心任務是接收一段未完成的句子，然後輸出字典裡每一個 Token 接在後面的機率分佈，藉此預測「下一個 Token 應該是什麼」**。

延續我們前面提到的「文字積木」概念，從你輸入一段話到模型產生回答，LLM 內部的運作可以拆解成以下幾個主要步驟：

**1. 將文字切分為 Token (Tokenization)**
首先，輸入的句子會先被切分成一個個的 Token，並各自對應到一個專屬的數字編號（ID）。

**2. 查表轉換為高維度向量 (Token Embedding)**
因為電腦無法直接處理這些數字 ID，模型內部會有一個龐大的「嵌入矩陣（Embedding Table）」。每個 Token 的 ID 都會透過這張表，被轉換成一排數字，也就是一個高維度的向​​量。在這個數學空間中，**意義相近的 Token 所對應的向​​量距離會非常接近**（例如「Apple」會跟「banana」或「iPhone」很接近）。

**3. 層層網路推敲上下文 (Layers 運作)**
轉換好的向​​量會進入模型內部的「層（Layers）」，這也就是所謂的深度學習（Deep Learning）或神經網路（Neural Network）的核心所在。大型模型通常有多個 Layer（例如 Llama 3B 有 28 層）。每一層主要會運用兩種機制來反覆處理這些向量：
*   **自注意力機制 (Self-Attention)：** 這是模型能夠「理解上下文」的關鍵。模型會去計算目前考量的 Token 與句子中前面其他 Token 之間的關聯程度（影響力）。例如輸入「兩顆青蘋果」，注意力機制會發現「青」和「兩」都會對「果」的意義產生不同面向的影響，並將這些資訊揉合在一起。在此過程中，模型也會加入「位置嵌入（Positional Embedding）」來分辨字詞的先後順序跟距離。
*   **前饋神經網路 (Feed Forward Network)：** 過完注意力機制後，向​​量會再經過一連串的矩陣乘法與活化函數（Activation Function）處理，做進一步的轉換。

經過這層層處理後，原本只代表單一字義的 Token 向​​量，就會變成**融合了整個句子上下文語意的向​​量（Contextualized Embedding）**。

**4. 計算最終得分 (LM Head)**
當向​​量跑完所有的 Layer 後，模型會拿出最後一層輸出的向​​量，將其乘上一個被稱為「LM Head」的矩陣（在許多語言模型中，這個矩陣其實跟一開始的 Embedding Table 是同一個）。這個過程等於是拿最終考慮好上下文的向量，去跟字典裡所有的 Token 向​​量計算「相似度」來打分數。**相似度越高的 Token，代表它越適合接在未完成的句子後面**。

**5. 轉換為機率並「擲骰子」 (Softmax)**
LM Head 算出來的分數（稱為 Logits）可能會有正有負，還不能當作真實的機率來用。因此，最後會經過一個名為 **Softmax** 的數學操作，將這些分數轉換成介於 0 到 1 之間、且總和為 1 的機率分佈。在此階段，系統也可以透過調整「溫度（Temperature）」參數，來決定機率分佈要越平均（讓模型偏向天馬行空的創意模式），還是越集中在少數幾個高分的字詞上（偏向保守精準）。

總結來說，LLM 就是將文字拆成積木、轉為數字向量後，在神經網路中透過「注意力機制」層層疊加對上下文的理解，最後算出每一塊積木接在後面的機率，以極度複雜的數學運算來玩「文字接龍」。

#### 可以用簡單的概念解釋 或是範例來舉例嗎

延續前面「文字積木」的概念，我們可以把大型語言模型（LLM）的運作想像成一座**「超大型的文字加工廠」**，而這座工廠唯一的工作目標，就是根據你給的未完成句子來玩「文字接龍」，預測下一個積木（Token）應該是什麼。

我們可以透過以下幾個平易近人的比喻與範例，來理解它內部的運作流程：

**1. 數十層的神經網路 ＝「流水線上的加工站」**
語言模型內部由非常多個「層（Layers）」組成，你可以把這些層想像成是**工廠流水線上的一個個站點**。每一塊文字積木進入工廠後，會一站一站地往下傳遞，每一站都會做一點處理，合起來就能完成非常複雜的語意理解。

**2. 注意力機制（Attention） ＝「員工左顧右盼，綜合判斷意思」**
在流水線的站點裡，有一個關鍵機制叫注意力機制。積木的意思往往不能單獨看，必須看上下文，所以負責處理的員工會去觀察「哪些其他的積木會影響當前積木的意思」：
*   **範例（觀察形容詞）：** 假設輸入的句子是「兩顆青蘋果」。如果只看最後一個「果」字，模型其實不知道它指的是西瓜、鳳梨還是蘋果。這時，站點裡的員工（注意力機制）會發現前面的「青」和「蘋」會深刻影響「果」的涵義，把這些資訊揉合在一起後，模型就知道這不是普通的果，而是「青色的蘋果」。
*   **範例（多個面向的觀察）：** 除了看顏色和種類，工廠裡還會有另一批專責的員工（也就是多頭注意力機制 Multi-head Attention），他們專門負責看「數量」。這批員工就會把注意力放在最前面的「兩」這個積木上，讓模型知道這是一個複數的詞彙。

**3. 考慮上下文的向量 ＝「同一個字，會隨情境變身」**
因為有了這條會互相參考的流水線，模型就能夠精準區分「長得一模一樣，但意思不同」的字。
*   **範例（區分 Apple）：** 假設有兩個句子，一句是「早餐吃了一顆 Apple（食物）」，另一句是「製造 iPad 和 AirPods 的 Apple（蘋果電腦）」。一開始剛進工廠時，這兩個 Apple 的積木長得一模一樣。但隨著它們在流水線上一層層往下走，經過員工對上下文的解讀，這兩個 Apple 在模型內部轉換成的模樣就會越來越不相像，最終模型就能完全明白它們是截然不同的兩個東西。

**4. 逐層解讀 ＝「偷窺模型的內心思考過程」**
這條流水線不只在加工，科學家甚至能透過技術，看到模型在每一層站點裡「內心的思考過程」。
*   **範例（中英轉換）：** 當我們輸入「天氣」兩個字，想讓模型接龍時，科學家發現模型在走到流水線中途的站點時，心裡其實是先浮現出英文的「weather」（天氣）與「forecast」（預報），直到經過最後一站，才決定配合使用者的語言，轉換成中文的「預」跟「報」，最終吐出「天氣預報」。
*   **範例（翻譯的內心戲）：** 當要求模型把法文的「fleur」（花）翻譯成中文時，模型在最開始幾層會吐出亂七八糟的字，但在中間幾層它其實會先想出英文的「flower」，最後才轉換並輸出中文的「花」。

**5. 轉換為機率 (Softmax) ＝「最後的擲骰子遊戲」**
當積木走完全部的流水線站點，來到最後一關時，模型會對字典裡幾萬個積木進行打分，並轉換成機率。最後，系統會根據這些機率來**「擲骰子」**，決定下一個要派出的積木是哪一個。
*   我們在使用語言模型時常聽到的「溫度（Temperature）」參數，就是在控制這顆骰子。如果把溫度調高，骰子的機率分佈會比較平均，模型就有機會擲出比較罕見、奇特的字，這就是很多模型標榜的**「創意模式」**；反之則會給出最保守、最常見的答案。

## agent 的概念是什麼？  它的原理 ？如何使用

延續我們前面聊過關於大型語言模型（LLM）和 AI Agent 的基礎概念，根據您提供的資料，我們可以從「概念」、「原理」與「如何使用」三個面向，更深入且具體地了解 Agent 的實際運作方式：

### 1. Agent 的概念是什麼？
傳統的大型語言模型（例如 GPT-4 等）雖然擁有強大的邏輯推理與問答能力，但它們有一個致命的限制：**無法主動感知或改變外部環境**。舉例來說，你可以請模型幫你寫出一個貪吃蛇遊戲的程式碼，但它沒辦法自己把這些程式碼儲存成電腦裡的檔案，還是需要你手動複製貼上。

為了解決這個問題，我們給語言模型接上各種「工具」（例如讀寫檔案的工具、執行終端機指令的工具、網路搜尋的工具），這些工具就像是模型的「感官和四肢」。**將「大模型」與「工具」組裝起來，讓它變成能夠自主感知並改變外在環境的智能程式，這就是 AI Agent**。例如知名的寫程式工具 Cursor，就是一個專門用於寫程式的 Agent。

### 2. Agent 的運作原理是什麼？
Agent 的運作是透過特定的「模式」來驅動模型自主解決問題。資料中介紹了兩種最核心的運作模式：

*   **ReAct 模式 (Reasoning and Acting, 思考與行動)：**
    這是目前最廣泛使用的 Agent 模式。它的核心原理是一個不斷循環的迴圈，包含四個關鍵步驟：**思考 (Thought) -> 行動 (Action) -> 觀察 (Observation) -> 最終答案 (Final Answer)**。
    當 Agent 收到任務時，大模型會先「思考」該怎麼做，接著「請求行動」（例如要求調用寫入檔案的工具）；Agent 的主程式會代為執行工具，並把工具執行的結果當作「觀察」回傳給大模型。模型根據新的觀察再次思考、行動，不斷循環，直到任務完成並給出「最終答案」。
*   **Plan and Execute 模式 (先規劃再執行)：**
    這個模式會將大任務拆解。它內部通常包含了負責擬定步驟清單的「規劃模型 (Plan 模 型)」、負責執行單一步驟的「執行 Agent」，以及負責檢視結果的「重新規劃模型 (Replan 模 型)」。模型會先列出完整的執行計畫，每執行完一步，就會根據新獲得的資訊（如查到的資料）來動態修改後續的計畫，直到所有步驟都走完。

### 3. 如何構建與使用 Agent？
*   **在底層構建上（系統提示詞的魔法）：**
    大模型之所以知道要「先思考再行動」，其實是依靠**「系統提示詞 (System Prompt)」**來規範的。開發者會寫一段類似劇本的指令給模型，規定模型必須使用特定的標籤（例如 `<thought>` 和 `<action>`）來輸出文字，並在提示詞中提供可用的工具列表與環境資訊給模型參考。主程式會解析這些標籤，替模型去執行真正的工具指令。
*   **在實際使用上：**
    對使用者來說，操作 Agent 非常輕鬆。**你只需要提供一個最終任務**。例如輸入指令：「寫一個貪吃蛇遊戲，使用 HTML、CSS 和 JS 實現，並將程式碼分別放在不同的檔案中」。
    接收到任務後，Agent 就會在背景全自動運轉。它會自己思考、自己請求呼叫工具寫入 index.html、確認寫入成功後再寫入 CSS 檔案，反覆操作直到整個遊戲建立完成。整個過程中，使用者基本上不需插手，只需等待 Agent 給出任務完成的最終回報即可。


## 什麼是 mcp 

綜合我們剛才聊到的 AI Agent 概念與資料，我們可以將 **MCP（Model Context Protocol，模型上下文協議）** 簡單理解為 AI 界的**「Type-C 通用接孔」**，它是用來完美連接 Agent 與各種外部工具的橋樑。

以下為您統整 MCP 的概念，以及它如何跟我們剛才聊到的 AI Agent 完美結合：

### 1. 為什麼會出現 MCP？（解決了什麼痛點）
我們剛才提到，Agent 的核心是「大語言模型（大腦）＋外部工具（感官與四肢）」，透過調用工具（例如搜尋天氣、讀寫檔案的函數），模型才能感知與改變真實世界。
然而，在工程實作上過去遇到了一個大麻煩：每一個 AI 平台（如 OpenAI、Anthropic、Google）要接上工具的「規格標準」都不一樣。這導致開發者寫好一個超棒的工具後，必須為每個平台各改寫一套專屬的串接程式碼，非常麻煩且沒有效率。

### 2. MCP 的概念是什麼？
MCP 就是為了解決上述痛點而誕生的**「統一工具介接標準協議」**。
就像現在的手機和筆電大多統一使用 Type-C 接孔一樣：有了 MCP 這套統一標準後，工具開發者只需要按照 MCP 的規範寫一次程式碼，這個工具就能隨插即用，直接套用在所有支援 MCP 的 Agent 系統或平台上，不需要重複開發。

### 3. 總結：大模型、Agent、工具與 MCP 的完整協作藍圖
把我們今天聊過的所有底層邏輯串聯起來，就可以得到一個「AI 數位員工」的完整運作架構：

*   **LLM 大語言模型（大腦）：** 負責玩高難度的「文字接龍」，理解使用者的最終任務並思考該怎麼做（例如我們前面提到的 ReAct 模式中，決定先思考再行動）。
*   **Tool / 外部工具（四肢與感官）：** 實際上去與外部世界互動的程式（例如「定位工具」獲取座標、「天氣工具」查詢氣象）。
*   **MCP（萬用關節）：** 一套標準化的通訊協議，讓大腦可以無縫接軌地插上並指揮各種不同來源的神兵利器，不需要擔心規格不合。
*   **AI Agent（自主數位員工）：** 一個結合了上述元素，能自主規劃（Plan）、透過 MCP 調用工具（Action）、觀察結果（Observation）並持續運作直到解決使用者問題的智能程式。

**💡 進階操作補充：Agent Skill（給員工的專屬操作手冊）**
有了裝上 MCP 工具的強大 Agent 後，如果你希望這個 Agent 完全照你的個人習慣做事（例如：要求它在幫你查完天氣後，必須根據下雨機率自動判斷要不要帶傘，而且回答必須用條列式），你可以給它一份稱為 **Agent Skill** 的 Markdown 說明文件。Agent 在執行任務時會先讀取這份說明書，接著嚴格按照你規定的步驟去調用對應的 MCP 工具，最終給出完美符合你格式與習慣的最終答案。

簡而言之，**Agent 是能自主規劃的機器人，工具是它的手腳，MCP 是裝上手腳的萬用卡榫，而 Agent Skill 則是這台機器人的專屬個人化操作手冊**。


## 什麼是 agent skill ？ 它的編寫規則是什麼？ 要如何開始第ㄧ步編寫 agent skill？

**Agent Skill 本質上就是你提前寫好給 Agent 看的一份「專屬操作說明書」**。它通常是一份 Markdown 格式的文檔，用來規定 Agent 處理特定任務時的具體步驟、判斷規則與輸出格式。

雖然 AI Agent 已經具備自主規劃和調用工具的能力，但它並不知道使用者的「私人習慣」與「特定要求」。假設你希望 Agent 成為出門小助手，每次出門前幫你看天氣，並規定它：「如果下雨要提醒帶傘、風大要提醒穿外套，而且回答必須先總結，再用條列式列出物品清單」。如果沒有 Agent Skill，你每次提問都必須輸入一大串重複的要求，非常反人類；而 Agent Skill 就是為了解決這個痛點而生。

一份完整的 Agent Skill 文檔通常包含兩個主要結構：

*   **元數據層（Metadata）：** 這相當於說明書的封面，用來告訴 Agent 這項技能的名稱（Name）與功能描述（Description）。
*   **指令層（Instructions）：** 這是核心的內容區域，用來寫明具體的目標、執行步驟（例如規定先調用定位工具，再調用天氣工具）、判斷規則、嚴格的輸出格式，以及提供給 Agent 模仿的對話範例。

**Agent Skill 的實際運作邏輯（以 Claude Code 為例）：**
1.  **存放與辨識：** 使用者需要將寫好的 Markdown 文檔，以特定檔名（例如必須嚴格命名為首字母大寫的 `Skill.md`）存放到系統指定的目錄中。
2.  **按需加載：** 系統啟動時，只會先讀取「元數據層」來了解有哪些技能可用。只有當你的提問（例如：「我要出門了，告訴我要帶什麼」）與某個技能的名稱和描述高度相關時，Agent 才會去讀取並載入完整的「指令層」細節，這種「漸進式披露機制」可以節省大量的 Token 消耗。
3.  **完美執行：** 讀取完畢後，Agent 就會嚴格依照說明書上的步驟，依序請求調用對應的 MCP 工具（如定位和天氣工具），經過內部規則判斷後，給出完全符合你個人習慣與格式要求的最終答案。

除了上述基礎應用，Agent Skill 還能支援運行程式碼、引用資源等更高階的功能，讓 Agent 徹底變成完全客製化的數位助理。

編寫一份高品質的 Agent Skill 說明文檔，本質上就是用 Markdown 格式為 Agent 寫一份**專屬操作說明書**。根據資料，一份完整且有效的 Agent Skill 文檔應該包含兩個主要結構，並遵循嚴格的檔案命名與存放規則：

### 一、 建立兩大核心結構

**1. 元數據層（Metadata，相當於說明書的封面）**
這部分用來告訴 Agent 這項技能的基本資訊，至少必須包含兩個屬性：
*   **Name（名稱）：** 定義這項技能的名字，例如「go out checklist」。
*   **Description（描述）：** 簡述這個技能的用途。系統啟動時會先只讀取這層資訊，當使用者的提問與名稱或描述高度相關時，才會去載入下方完整的指令層，這種**「漸進式披露機制」能有效節省 Token 的消耗**。

**2. 指令層（Instructions，核心內容區域）**
這部分沒有嚴格的語法格式規定，**唯一的重點在於把事情向 Agent 說明明白**。一份高品質的指令層建議包含以下要素：
*   **目標（Goals）：** 明確指出這個 Skill 要完成的最終目的。
*   **執行步驟（Steps）：** 清楚條列 Agent 該做什麼。例如規定它「先調用定位工具獲取經緯度，然後再調用天氣工具獲取天氣資訊」。
*   **判斷規則（Rules）：** 給予 Agent 處理資訊的邏輯。例如告訴它如何根據天氣數據，判斷出門需要帶什麼物品。
*   **輸出格式（Output Format）：** 嚴格規定 Agent 回答的排版要求。例如要求必須輸出兩段話，先給出總結，再列出清單。
*   **示例（Examples）：** 提供標準對話範本給 Agent 模仿。範本中應包含「假設的用戶提問」、「工具返回的模擬數據」，以及「期待 Agent 輸出的最終結果樣貌」。

### 二、 遵守嚴格的存放與命名規範

就算文檔內容寫得再好，如果沒有按規矩存放，系統也無法辨識。以 Claude Code 為例，寫好的文檔必須遵循以下存檔規則：
1.  **資料夾名稱必須與 Skill 名稱完全相同**：在你指定的存放目錄下建立新資料夾，其名稱必須跟你設定的 `name` 一模一樣（例如 `go out checklist`）。
2.  **檔案名稱必須嚴格命名為 `Skill.md`**：在該資料夾內把內容貼上並建立檔案，**檔名必須是 `Skill.md`（注意首字母 S 必須大寫）**。這是系統辨識的「接頭暗號」，隨便亂取別的名字系統是絕對不會認的。

## 何時會使用skill 

要區分何時使用 MCP、何時使用 Agent Skill，可以把握一個核心概念：**MCP 是用來擴充 Agent 的「能力」（提供工具），而 Agent Skill 是用來定義 Agent 的「行為準則與個人化習慣」（提供說明書）**。

以下是這兩者的具體使用時機與差異：

**一、 何時會使用 MCP (Model Context Protocol)？**
當你需要**讓大語言模型感知或改變外部真實世界**，也就是需要給它裝上「感官和四肢」來執行動作時，就會使用 MCP。
*   **跨平台提供統一的工具介面：** 如果你開發了一個能查詢天氣、定位或是操作電腦檔案的程式（工具），為了避免要為 ChatGPT、Claude、Gemini 各寫一套專屬的串接程式碼，你可以遵守 MCP 這套「統一工具介接標準協議」來開發。這就像是打造一個 Type-C 接頭，只要寫一次程式碼，所有支援 MCP 的平台都能直接隨插即用。
*   **賦予 Agent 解決新問題的能力：** 當你發現目前的 Agent 無法獲取外部資訊（例如無法得知即時天氣、無法獲取當地經緯度）時，你需要引入對應的 MCP 工具來擴充它解決複雜問題的能力。

**二、 何時會使用 Agent Skill？**
當你已經有了具備各項工具能力的 Agent，但你**希望它在執行特定任務時，能嚴格遵守你的私人習慣、判斷邏輯與輸出格式**，以省去每次提問都要輸入長串提示詞的麻煩時，就會使用 Agent Skill。
*   **將複雜且重複的步驟標準化：** 例如你希望 Agent 成為「出門小助手」，如果沒有 Skill，你每次提問都必須打一大段話提醒它「先查定位、再查天氣、最後列出清單」。有了 Agent Skill，你可以提早寫好一份 Markdown 說明文件，規定好它遇到這類任務時的標準作業程序（SOP）。
*   **客製化專屬的判斷邏輯與排版：** 當你需要 Agent 套用特定規則時（例如：規定「下雨必須帶傘」、「光照強帶帽子」，且要求回答必須「先總結，再列出原因和物品清單」），Agent Skill 裡的指令層能強迫 Agent 完美模仿並遵守你的個人化要求。

**總結：兩者如何完美搭配？**
在實際應用中，MCP 和 Agent Skill 通常是互補合作的。以打造「專屬出門小助手」為例：
你必須先**透過 MCP 匯入「定位工具」與「天氣工具」**，讓 Agent 在底層具備查詢位置和氣象的「能力」；接著，你再**寫一份 Agent Skill 給 Agent 看，告訴它當用戶說「我要出門了」時，必須先呼叫定位工具，再呼叫天氣工具**，最後根據查到的天氣數據，按照你的私人習慣整理出一份完美的出門清單。
