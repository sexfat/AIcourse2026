主題 01

傳統程式 vs 生成式 AI：根本的典範轉移

傳統程式

規則驅動

輸入資料 ＋ 工程師寫死的規則（If/Else、演算法）＝ 輸出結果。
工程師扮演「上帝」，所有邏輯與邊界條件都必須明確定義；漏寫一個 if，系統就會噴 Bug。

生成式 AI

機率驅動

餵入海量資料 ＋ 模型訓練 ＝ 機器自己找出潛在規則。
工程師不再寫死規則，而是給予「提示與上下文（Prompt）」，AI 基於統計規律去「憑空創造」新內容。

一句話區分：傳統 AI 拿來做分類與判斷（垃圾信偵測、圖片辨識）；生成式 AI 拿來創造新東西（寫程式、畫圖、生成報告）。

生成式 AI 的本質是「超級文字接龍機」，根據你給的前文，計算下一個字或像素「最有可能出現的是什麼」，進而組合出程式碼、圖片或文章。

主題 02

Token：語言模型的「文字積木」

AI 看不懂人類的完整句子。它必須先把文字切解成一塊塊的 Token，轉成數字才能進行數學運算。Token 是生成式 AI 計算成本與理解語言的最基本單位。

工程師隱喻
把 Token 想像成「文字積木（Lego 積木）」。每塊積木有專屬編號（Token ID），意義相近的積木在數學空間中距離相近，而語言模型的工作就是——根據目前擺好的積木組合，「猜」下一塊最合適的積木。

Token 的四大特性

  • 切法不等於一個字：「Apple」、「APPLE」、「 Apple」（前面有空格）是三個不同 Token；中文有時數字合成一個 Token，有時一字被切成多塊；標點符號也是獨立 Token。
  • 每個 Token 有唯一 ID：Llama 模型中，編號 0 的 Token 就是「！（驚嘆號）」。電腦只懂數字，因此所有文字最終都被轉成整數 ID。
  • ID 再轉為高維度向量（Embedding）：透過 Embedding Table「查表」，每個 Token ID 轉成一排帶有語意的數字向量。意義相近的字（如 Apple、iPhone、蘋果）在向量空間中距離相近。
  • Context Window = AI 的工作記憶：Token 數超過上限，AI 就開始「忘記」前面的對話。這就是為什麼長文輸入需要分段處理。
為什麼計費用 Token？因為 Token 才是 AI 實際運算的單位，不是字數。英文通常 1 個詞 ≈ 1–2 Token；中文約 1–2 字 ≈ 1 Token（視模型而異）。

積木的語意範例

Apple 的向量鄰居

水果面：banana、mango
科技面：MacBook、iPhone、Cupertino
中文面：蘋果

模型依上下文判斷你說的是哪個意思。

文字接龍範例

輸入「兩顆青蘋」→ 模型預測「果」機率最高
輸入「天氣」→ 模型內部先想到 weather/forecast，再輸出「天氣預報」

主題 03

LLM：大型語言模型的運作原理

大型語言模型（LLM，如 GPT-4、Claude）本質上是一個複雜的數學函式，核心任務是接收未完成的句子，輸出每個 Token 接在後面的機率分布，藉此預測下一個字。

工程師隱喻
把 LLM 想像成一座「超大型文字加工廠」，有數十條流水線（Layers），唯一目標是根據你給的句子玩「文字接龍」——預測下一塊積木。

5 道工序

① Tokenize句子切成 Token 積木，轉成數字 ID
② Embed每個 ID 查 Embedding Table，轉成高維向量
③ Layers向量進入多層神經網路，注意力機制理解上下文
④ LM Head最終向量跟字典所有 Token 計算相似度打分數
⑤ Softmax分數轉機率，「擲骰子」選出下一個 Token

關鍵機制詳解

機制 作用 範例
Self-Attention（自注意力） 計算每個 Token 與其他 Token 的關聯強度，融合上下文語意 「兩顆青蘋果」中，「青」和「兩」都影響「果」的含義
Feed Forward Network 向量經過矩陣乘法與激活函數進一步轉換 每一層都讓模型「理解更深」
Contextualized Embedding 同一個字依上下文產生不同向量 「Apple（水果）」vs「Apple（公司）」最終向量截然不同
Temperature（溫度） 控制 Softmax 的機率分布集中程度 高溫 → 創意模式；低溫 → 保守精準
科學家「偷窺」模型的思考過程：當輸入「天氣」，模型在中間層其實先浮現英文 weather / forecast，直到最後一層才轉換為中文「天氣預報」輸出。翻譯時也是先在內部過英文再輸出目標語言——這就是多語言模型的運作方式。
主題 04

Prompt Engineering：工程師的新型態程式語言

過去

用程式語言下指令

Python / JavaScript 對機器說「做這件事」，邏輯精確、沒有歧義，錯一個字就報錯。

現在

用自然語言下指令

Prompt 是一種「收斂機率分布」的工具。Prompt 越模糊，AI 自由發揮空間越大，越容易「幻覺」；越精確，輸出越可預期。

好 Prompt 的 5 個欄位

1. 角色你是一個資深前端工程師
2. 背景目前狀況與相關檔案
3. 限制不能改什麼、偏好什麼
4. 輸出要程式、摘要、表格或步驟
5. 驗收如何確認任務完成
Prompt 模板
你是一個 [角色]。背景是 [目前狀況]。
請幫我 [目標]，限制是 [限制條件]。
完成後請 [輸出格式]。驗收方式是 [測試或檢查方式]。

常見陷阱 vs 正確做法

避免
  • 「幫我做一個 App」（過於模糊）
  • 沒說明限制條件和禁止事項
  • 沒要求驗收步驟
  • 把 API Key 或密碼直接寫進 Prompt
改成
  • 明確說明角色、背景、目標、格式
  • 列出不能碰的檔案或函式
  • 要求提供測試指令與手動驗收步驟
  • 用佔位符（如 $API_KEY）代替機密
主題 05

AI Agent：讓 AI 從「回答者」變成「執行者」

傳統 LLM 的限制：能寫出貪吃蛇程式碼，但無法主動把程式碼存成檔案，還是需要你手動複製貼上。

Agent 的定義：把「大語言模型（大腦）」接上「工具（感官與四肢）」，讓它能自主感知並改變外部環境的智能程式，就是 AI Agent。工具包括：讀寫檔案、執行終端機指令、呼叫 API、瀏覽網頁等。

兩大核心運作模式

ReAct 模式（最廣泛）

思考 → 行動 → 觀察 → 循環

  1. 思考（Thought）：LLM 分析任務，決定下一步
  2. 行動（Action）：呼叫工具（如寫入檔案）
  3. 觀察（Observation）：工具結果回傳給 LLM
  4. 不斷循環直到任務完成，給出最終答案
Plan & Execute 模式

先規劃、再逐步執行

  • 規劃模型先制定完整步驟清單
  • 執行 Agent逐步完成每個步驟
  • 每完成一步，動態修改後續計畫
  • 適合大型、複雜任務的拆解

底層運作原理：System Prompt 的魔法

大模型之所以知道「先思考再行動」，是靠系統提示詞（System Prompt）規範的。開發者會寫一段劇本給模型，規定它必須用特定標籤（如 <thought> <action>）來輸出，主程式解析這些標籤後，代替模型執行真正的工具指令。

使用者怎麼用 Agent
「寫一個貪吃蛇遊戲，使用 HTML、CSS 和 JS，
程式碼分別放在不同的檔案中。」

Agent 會自行思考、建立檔案、確認成功、再建下一個，全程不需插手。

主題 06

MCP：AI 工具的「Type-C 通用接孔」

過去的痛點

工具規格不統一

每個 AI 平台（OpenAI、Anthropic、Google）接上工具的方式都不同。開發者做好一個好用的工具，必須為每個平台各寫一套串接程式碼，極度低效。

MCP 的解法

Model Context Protocol

統一工具介接標準。就像手機充電統一使用 Type-C：工具只需按 MCP 規範寫一次，所有支援 MCP 的 Agent 平台都能「隨插即用」。

完整的 AI 數位員工架構

LLM（大腦）理解任務、思考步驟、決定行動
Tool 外部工具（四肢）查天氣、讀寫檔案、呼叫 API
MCP（萬用卡榫）標準化通訊協議，讓大腦無縫插上各種工具
AI Agent（數位員工）自主規劃、執行、觀察，直到任務完成
對工程師的實際意義：如果你要開發工具供 AI 使用，遵守 MCP 規範，你的工具就能同時在 Claude、ChatGPT、Gemini 等所有支援 MCP 的平台上運作，不需要重複開發。
主題 07

Agent Skill：給 Agent 的專屬操作說明書

Agent 有了工具能力（MCP），但不知道你的私人習慣與格式要求。每次提問都要輸入一大串重複的要求，非常低效。

比喻
Agent Skill 就像給新員工的「SOP 手冊」。你把步驟、判斷規則、輸出格式都寫好，員工（Agent）拿到手冊後，以後執行同類任務就會嚴格照手冊走，不需要再解釋。

一份完整 Agent Skill 的兩大結構

元數據層（Metadata）

說明書的封面

  • 寫在檔案最上方的 YAML frontmatter（--- 包起來的區塊）
  • name：技能名稱（如 go-out-checklist）
  • description：功能描述，讓系統判斷何時載入

系統啟動只讀 Metadata，用到才載入完整指令，節省 Token。

指令層（Instructions）

核心內容區

  • Goals：最終目的
  • Steps：先呼叫哪個工具、再做什麼
  • Rules：判斷邏輯（如「下雨才提醒帶傘」）
  • Output Format：嚴格規定輸出排版
  • Examples：標準對話範本供 AI 模仿

存放規則（以 Claude Code 為例）

  • 資料夾名稱必須與 name完全相同（如 go-out-checklist/）
  • 檔案必須命名為 SKILL.md（全部字母大寫，不是只有首字母）
  • 檔案開頭必須有 --- 包起來的 YAML frontmatter，內含 name 與 description
  • 放到系統指定目錄（如 ~/.claude/skills/），Agent 啟動時自動偵測

MCP vs Agent Skill 的使用時機

MCP Agent Skill
目的 擴充 Agent 的「能力」（提供工具） 定義 Agent 的「行為準則」（提供說明書）
類比 給員工裝上電話、電腦、車子 給員工一本 SOP 手冊
使用時機 需要 Agent 感知或改變外部世界時 需要 Agent 執行重複性、有固定格式要求的任務時

官方工具：Skill Creator —— 一站式建立 Skill

手寫 SKILL.md 最容易出錯的地方，往往不是內容想不清楚，而是 frontmatter 格式、資料夾命名這類「規矩」。Anthropic 官方推出了 Skill Creator，本身就是一個 Skill——專門用來引導你產生其他 Skill，內建在 Claude（claude.ai / Claude Desktop / Claude Code）裡可以直接呼叫。

比喻
Skill Creator 就像請一位熟悉公司規範的資深同事，幫新手寫 SOP 手冊——你只要把想法講清楚，他會處理格式、命名、目錄結構這些瑣碎但容易出錯的細節。

Skill Creator 內建的三個工具

init_skill.py

建立骨架：產生符合規範的 SKILL.md 模板，以及 scripts/、references/、assets/ 範例目錄。

quick_validate.py

驗證格式：檢查 frontmatter 是否存在、YAML 語法是否正確、必填欄位是否齊全。

package_skill.py

打包發布：把整個資料夾壓縮成一個 .skill 檔，方便分享或上傳給其他人安裝。

如何使用

① 開口描述跟 Claude 說「幫我用 skill-creator 建立一個 OO 的 Skill」，描述用途與典型情境
② 互動釐清Claude 會反問細節（觸發情境、步驟、輸出格式），協助把 description 寫得夠具體
③ 產生與填寫執行 init_skill.py 產生骨架，再依 TODO 提示把 Overview、步驟、規則寫完整
④ 驗證與打包用 quick_validate.py 檢查格式，沒問題再用 package_skill.py 打包成 .skill 檔
小提醒：Skill Creator 產生的只是「骨架」與「格式檢查」，指令層裡 Goals/Steps/Rules 寫得夠不夠精準，還是要靠你對任務本身的理解——工具能幫你不犯格式上的低級錯誤，但不能取代想清楚「這個 Skill 到底要解決什麼問題」。
主題 08

Claude Code & Codex：兩大 AI 開發工具

Claude Code

本機終端機 AI 協作者

  • Read：瀏覽資料夾、閱讀檔案、理解程式結構
  • Change：編輯程式碼、文件、設定檔，沿用現有風格
  • Verify：執行指令、跑測試、看錯誤訊息、回報結果

終端機輸入 claude 即可啟動。把它當成會問問題、會動手的 Junior Pair。

Codex

OpenAI 雲端 AI 工程師

  • 雲端沙箱環境，任務可平行執行
  • 支援 GPT 系列模型
  • 適合需要隔離環境或大量平行任務的場景

兩者可互補使用，選擇適合工作情境的工具。

共同特點：兩者都能讀懂整個專案架構、修改程式碼、執行指令，並支援自然語言（Prompt）操作。差異主要在於本機 vs 雲端、即時互動 vs 平行執行。

任務 A

用 AI 快速了解陌生專案 + 找出技術債

Debug 練習環境

開啟練習 ↗
目標：在不熟悉新專案的狀態下，利用 AI 快速釐清程式脈絡，找出問題點（過期寫法、命名不一致），並整理成改進建議報告。
建議工具：Claude Chat / Claude Code / Codex  |  預計時間：30 分鐘

Step 1：了解陌生專案架構

Prompt 範例
請先閱讀這個專案，不要修改任何檔案。

請回覆：
1. 專案用途是什麼？
2. 主要資料夾與檔案結構
3. 程式入口點在哪？
4. 本機如何啟動？
5. 你建議我先看哪 3 個檔案？

Step 2：找出程式碼問題點

要找的問題類型
  • 技術債：過期寫法、冗餘程式碼、函式過長
  • 安全漏洞：未驗證輸入、硬編碼密碼、SQL Injection
  • 命名不一致：camelCase / snake_case 混用，語意不明
Prompt 範例
請分析這個專案，找出：
- 技術債或過期寫法
- 潛在安全風險
- 命名不一致問題

以條列格式整理，
並標明位於哪個檔案。

Step 3–4：整理報告 & 請 AI 改善程式碼

Step 3列出所有問題，按嚴重程度排序
Step 4選擇優先問題，要求 AI 提供改善建議
Step 5請 AI 提供修改後程式碼並說明理由
Step 6審查 Diff，確認修改範圍符合預期再接受
記住：AI 提供的程式碼需要你來審查。看 Diff、問理由、驗收測試是工程師的責任，不能全盤接受。
任務 B

用 AI 生成「卡路里紀錄 Dashboard」

UI Design 練習環境

開啟練習 ↗
目標：使用結構化 Prompt，引導 AI 用單一 HTML 檔案（內嵌 Tailwind CDN + 原生 JS）生成具備動態互動功能的前端 Dashboard。
建議工具：Claude Chat / ChatGPT  |  預計時間：45–60 分鐘

UI 三大區塊

頂部狀態欄今日目標卡路里、已攝取、剩餘，以進度條顯示
搜尋區食物搜尋輸入框、熱門標籤、搜尋結果卡片清單
日記清單今日已加入食物清單，支援刪除與自動重新計算

結構化 Prompt 範例（生成 UI 佈局）

Prompt 範例
角色：你是一個資深的前端工程師與 UI 設計師。
任務：請使用單一 HTML 檔案（內嵌 Tailwind CSS CDN 與原生 JavaScript）
建立一個「美食搜尋與卡路里紀錄」的 Dashboard 介面。

版面佈局要求（RWD 行動優先）：
1. 頂部大盤：呈現今日卡路里儀表板，包含「目標 2000 kcal」、
   「已攝取 0 kcal」、「剩餘 2000 kcal」三個數值區塊，
   並包含一個橫向的視覺進度條。
2. 下方雙欄配置（桌面版 md:grid-cols-3）：
   - 左側欄（佔 2 欄）：美食搜尋區
   - 右側欄（佔 1 欄）：今日飲食日記清單

視覺風格：背景 slate-50，主按鈕與進度條使用 emerald-500 綠色系。

輸出限制：請先提供完整的 HTML 與 CSS 佈局結構，
JavaScript 邏輯先留白，我們稍後處理。

三個常見陷阱（工程師必讀）

  • 狀態管理混亂：AI 常在 click handler 裡同時修改 DOM 和計算數值，高度耦合難維護。要求 AI 採用「資料驅動（Data-Driven）」模式，所有操作只修改 state，由獨立 render() 函式統一刷新 UI。
  • Tailwind 動態類名失效：class="bg-${color}-500" 這種字串拼接在 Tailwind CDN 中完全無效。動態樣式要直接操作 element.style.width 或用完整類名切換（bg-red-500、bg-emerald-500）。
  • 進度條邊界條件（Edge Cases）：吃超過目標時，百分比會超過 100%，進度條爆框。要在邏輯層加上 Math.min(percent, 100)，超標時進度條自動變紅色警示。

規格驅動

SDD：先寫規格再寫程式——讓 AI 成為可靠的開發夥伴

Spec-Driven Development（SDD，規格驅動開發）的核心精神：不是直接叫 AI「寫 code」，而是要求它先把需求轉成規格文件、再做技術規劃、再拆任務，最後才實作。這樣做能大幅降低 AI 幻覺，並讓每個產出物都有可驗證的標準。

為什麼需要 SDD？直接叫 AI「幫我寫一個購物車功能」，它會做出「一個」購物車，但不一定是你要的那個。先寫 spec，讓 AI 把需求說清楚、找出模糊地帶，你確認後再動工，才不會浪費時間在錯誤的方向上。

三份核心文件：spec → plan → tasks

spec.md需求、功能範圍、驗收標準——搞清楚「要做什麼」
plan.md系統架構、API 合約、技術取捨——搞清楚「怎麼做」
tasks.md可執行任務清單，每個 task 有驗收條件，可直接交給 coding agent

spec.md 必備欄位

需求面
  • 背景與目標
  • 問題定義
  • In scope / Out of scope
  • 使用者角色
  • 使用情境 / user flow
  • 功能需求（每條需可測試）
  • 非功能需求
驗收面
  • 資料需求
  • API / integration 需求
  • 權限與安全需求
  • 驗收標準（Acceptance Criteria）
  • 風險與未決事項（Open Questions）
寫好 spec 的黃金守則：每一條需求都必須可測試。「介面要好看」「速度要快」這類模糊字眼，要改成「首頁載入時間 < 2 秒」「按鈕在 1024px 以上螢幕顯示為水平排列」等具體可驗證的條件。

SDD Prompt 模板（直接貼給 AI）

SDD Agent Prompt
你現在是我的 Spec-Driven Development AI Agent。
你的任務不是直接寫程式，而是先把需求整理成
可實作、可驗收、可追蹤的規格文件，再進入技術規劃與任務拆解。

請嚴格遵守以下流程：

1. 先理解需求
   - 摘要我提供的需求
   - 找出不清楚、矛盾、缺失的地方
   - 若資訊不足，先提出必要問題
   - 不要自行腦補商業規則

2. 產出 spec.md（等我確認後再繼續）
   背景與目標 / 問題定義 / In-Out scope /
   使用者角色 / user flow / 功能需求 /
   非功能需求 / 資料需求 / API 需求 /
   權限安全 / 驗收標準 / 風險與未決事項

3. 產出 plan.md（spec 確認後）
   系統架構 / 前後端方案 / 資料模型 /
   API contract / 狀態流與錯誤處理 /
   第三方整合 / 安全性 / 技術風險

4. 產出 tasks.md（plan 確認後）
   任務編號 / 名稱 / 描述 / 前置依賴 /
   產出物 / 驗收條件 / 優先級 / 是否可並行

實作守則：
- 沒有 spec，不開始 coding
- 沒有 acceptance criteria，不標示完成
- API 採 design-first，先定 request/response/schema
- 需求變更，先更新 spec → plan → tasks，不可直接改 code

請先執行第 1 階段：理解需求與提出澄清問題。
SDD 核心心法：「要做什麼」和「怎麼做」必須分開。spec.md 只談業務需求，plan.md 才談技術實作。讓 AI 先把問題說清楚再動工，這才是 AI-Native 開發的正確節奏。
協作心法

跟 AI 工具協作的基本節奏

Explore先讀懂專案，不急著改
Plan確認修改範圍與限制
Implement小步修改，保持可回滾
Verify跑測試或手動驗收
Summarize整理成果與潛在風險
推薦工作流：「先分析，不修改」→「提出方案」→「最小範圍實作」→「跑測試」→「總結 diff」
常見錯誤
  • 一次丟太大的任務，超出 AI 能掌控的範圍
  • 沒有說清楚限制條件（不能動哪些檔案）
  • 沒有要求驗證步驟，直接上線
  • 不看 Diff 就直接接受所有修改
  • 把 secret / API key 寫進 Prompt
正確做法
  • 拆成「探索 → 修改 → 驗證」三個步驟
  • 明確列出不能碰的範圍與偏好風格
  • 要求測試指令與手動驗收步驟
  • 看 Diff，請 AI 解釋每個修改的理由
  • 用環境變數或佔位符代替機密資訊
