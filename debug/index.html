學習目標

01
🗺️
快速理解陌生專案架構

03
📋
整理報告 + 產出改善程式碼
將 AI 分析結果整理成結構化問題報告，並請 AI 提供具體改善後的程式碼。

Prompt 範例

// copy & paste into Claude
Prompt 1 — 了解架構
以下是一個 Node.js + 原生 JS 的 Todo App。 請幫我： 1. 說明這個專案的架構與各檔案職責 2. 列出所有 API 路由與功能 3. 描述資料流（從前端 → 後端 → 資料庫） [貼上 server.js 和 app.js 的程式碼]
Prompt 2 — 找問題
請仔細審查以下程式碼，找出所有問題， 包括： - 安全漏洞（SQL Injection、硬編碼密鑰等） - 過時或不良的程式寫法 - 命名不一致的地方 - 未處理的錯誤 - 重複或無用的程式碼 請以表格方式整理，說明每個問題的 影響程度（高/中/低）與改善建議。 [貼上 server.js 和 app.js 的程式碼]
🐛

這個專案埋了什麼問題？

// expand to reveal answers
1
SQL Injection
直接用字串拼接組成 SQL 查詢，攻擊者可輸入惡意字串讀取或刪除全部資料。
應改用 ? 佔位符的 Prepared Statement。
2
硬編碼密鑰 + 測試路由洩漏
SECRET_KEY = 'abc123password' 寫死在程式碼中，且 /test 路由會直接將它回傳給任何人。
應改用 process.env.SECRET_KEY 並刪除測試路由。
1
全面使用 var
應改用 const（不變）或 let（可變），避免無塊作用域導致的非預期行為。
2
舊式 XMLHttpRequest
應改用現代 fetch API 搭配 async/await，程式碼更簡潔、可讀性更高。
3
== 寬鬆比較
應統一改用 === 嚴格相等，避免非預期的型別自動轉換錯誤。
1
user_name  vs  UserName
snake_case 與 PascalCase 混用，代表同一個「使用者名稱」欄位。
2
created_at  vs  createdAt
snake_case 與 camelCase 混用於同一個時間欄位。
3
getUserData  vs  get_user_info
功能相同的兩個函式，命名風格截然不同，還伴隨重複程式碼問題。
4
currentUser  vs  TodoList
全域變數一個用 camelCase，一個用 PascalCase，風格不一致。
5
var  /  let 混用
同一個檔案中部分用 var、部分用 let，缺乏統一的 coding style。
1
Closure 陷阱
for + var 搭配 closure，所有 checkbox 和刪除按鈕都只會操作最後一筆 todo。
應改用 forEach 或將 id 傳入屬性。
2
錯誤未回傳給前端
多個 API 路由發生錯誤時只做 console.log，卻繼續回傳 success: true，前端無從得知操作失敗。
3
Dead Code
unusedConfig、formatDate()、format_date_string() 宣告後從未被使用，增加維護負擔。
4
Callback 內 return 無效
get_user_info() 在非同步 callback 裡 return row，這個值永遠無法被呼叫方取得。
