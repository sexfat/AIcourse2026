// Afternoon Session — Writing Your Own Agent Skill

用 AI 寫一份 Agent Skill

親手打造一份「Code Review Checklist」Skill，體驗 Metadata / Instructions 兩層結構，並驗證它有沒有被 Claude Code 正確觸發。

Claude Code SKILL.md YAML Frontmatter 漸進式披露 Prompt Engineering
🎯

學習目標

// learning objectives
01
🗂️
理解兩層結構與存放規則
看懂 Metadata（YAML frontmatter）與 Instructions 的分工，並掌握 ~/.claude/skills/<name>/SKILL.md 的嚴格命名規則。
02
🧩
親手寫一份 Code Review Skill
把 Goals、Steps、Rules、Output Format、Examples 五個要素，寫成一份能實際運作的 SKILL.md。
03
🧪
驗證觸發與邊界情況
測試「該觸發時有沒有觸發」「不該觸發時有沒有誤觸發」「沒問題時會不會硬掰問題」三種情境。
📝

練習步驟

// follow these steps
01
先看一份真實範例 START HERE
在終端機執行 cat ~/.claude/skills/git-commit-summary/SKILL.md，觀察真正在運作的 Skill 長什麼樣子：YAML frontmatter、資料夾名稱與 name 一致、全大寫的 SKILL.md 檔名。
02
設計 Metadata 層
決定 name（小寫、用 - 連接，如 code-review-checklist）與 description。description 要寫清楚「使用者會說什麼話」，這是系統判斷何時載入的唯一依據。
03
設計 Instructions 層五要素
依序寫出 Goals、Steps、Rules（含嚴重度判斷邏輯）、Output Format（固定的輸出結構）、Examples（一組輸入 → 期待輸出）。參考下方「起始模板」。
04
依規則建立資料夾與檔案
mkdir -p ~/.claude/skills/code-review-checklist，再把內容存成 SKILL.md（全部大寫，不是 Skill.md）。
05
開新對話實際測試
開一個新的 Claude Code 對話視窗（Skill 在啟動時載入，舊視窗可能偵測不到）。分別測試：① 貼一段有問題的程式碼說「幫我 review」② 問一個不相關的問題，確認沒有被誤觸發。
06
進階：邊界情況 挑戰
貼一段完全乾淨、沒有問題的程式碼，確認 Agent 會老實說「沒有發現問題」，而不是硬掰一些無關緊要的意見來湊滿你規定的輸出格式。
🧩

SKILL.md 起始模板

// copy & fill in the blanks
起始模板 — SKILL.md
--- name: code-review-checklist description: 補上「什麼情境會用到」，例如：當使用者要求 code review、 審查 diff、檢查 PR 品質，或說「幫我看一下這段 code」時使用。 --- # Code Review Checklist ## Goals （這個 Skill 最終要幫使用者達成什麼？） ## Steps 1. 取得要審查的內容（使用者貼的程式碼，或 git diff） 2. 依 Rules 逐項檢查 3. 依 Output Format 整理結論 ## Rules （不同類型的問題，怎麼判斷嚴重程度？ 沒有發現問題時要怎麼處理？） ## Output Format （輸出必須長什麼樣子，才能一眼看出能不能合併？） ## Examples （給一組「假設輸入 → 期待輸出」的範例）
測試用程式碼（貼給你的 Skill）
幫我 review 一下這段 code： function getUser(id) { db.query("SELECT * FROM users WHERE id = " + id, (err, row) => { console.log(err); return row; }); }

進階：官方工具 Skill Creator

// the official skill-building skill

手寫 SKILL.md 最容易出錯的不是內容想不清楚，而是 frontmatter 格式、資料夾命名這類「規矩」。Anthropic 官方推出了 Skill Creator ↗——本身就是一個 Skill，專門用來引導你產生其他 Skill，內建在 Claude（claude.ai / Claude Desktop / Claude Code）裡可以直接呼叫，不需要額外安裝。

init_skill.py
🏗️
建立骨架
產生符合規範的 SKILL.md 模板，以及 scripts/、references/、assets/ 範例目錄。
quick_validate.py
✅
驗證格式
檢查 frontmatter 是否存在、YAML 語法是否正確、必填欄位（name/description）是否齊全。
package_skill.py
📦
打包發布
把整個資料夾壓縮成一個 .skill 檔，方便分享或上傳給其他人安裝。

如何使用（四步）

①
開口描述
跟 Claude 說「幫我用 skill-creator 建立一個 code-review-checklist 的 Skill」，描述用途與典型情境。
②
互動釐清
Claude 會反問細節（觸發情境、步驟、輸出格式），協助把 description 寫得夠具體。
③
產生與填寫
執行 init_skill.py 產生骨架，再依 TODO 提示把 Overview、Steps、Rules 寫完整。
④
驗證與打包
用 quick_validate.py 檢查格式，沒問題再用 package_skill.py 打包成 .skill 檔。
開場 Prompt 範例
幫我用 skill-creator 建立一個新的 Skill： 名稱方向：code-review-checklist 用途：使用者貼程式碼或要求 review 時，依固定檢查清單 （安全漏洞、錯誤處理、命名一致性、重複程式碼）審查， 並給出「可以合併 / 需要修正」的明確結論。 請引導我把 description 與 Rules 寫清楚， 再用 init_skill.py 幫我產生骨架。
CLI 指令參考
# 1. 建立骨架 python init_skill.py code-review-checklist --path ~/.claude/skills # 2. 驗證格式 python quick_validate.py ~/.claude/skills/code-review-checklist # 3. 打包成 .skill 檔（可選，用於分享） python package_skill.py ~/.claude/skills/code-review-checklist
小提醒：Skill Creator 產生的只是「骨架」與「格式檢查」，指令層裡 Goals/Steps/Rules 寫得夠不夠精準，還是要靠你對任務本身的理解——工具能幫你不犯格式上的低級錯誤，但不能取代想清楚「這個 Skill 到底要解決什麼問題」。建議先照本單元的手寫流程練過一輪，再用 Skill Creator 加速下一份 Skill。
延伸資源：不想從零開始？Skill Marketplace（skillsmp.com）↗ 收錄社群分享的現成 Skill，可以先逛逛別人怎麼寫，再回頭調整自己的版本。
⚠️

常見的撰寫疏失

// expand to reveal
1
description 寫太模糊
只寫「審查程式碼」會讓系統不確定何時該載入，或誤觸發在不相關的對話。要把使用者實際會說的話寫進去。
2
Rules 沒有覆蓋「找不到問題」的情況
沒有明確規定「沒問題就說沒問題」，Agent 會為了符合輸出格式硬掰出問題。
3
Output Format 沒有規定具體結構
沒有固定要不要表格、要不要先給結論，測試時很難判斷 Agent 有沒有真的照做。
4
資料夾名稱跟 name 沒有完全一致
多一個空格、大小寫不同，都會導致 Skill 沒被偵測到。
5
檔名打成 Skill.md 或 skill.md
規定是全大寫的 SKILL.md，大小寫不對系統就找不到這個檔案。
✅

完成後的驗收清單

// acceptance checklist
SKILL.md 存放在 ~/.claude/skills/<name>/SKILL.md，資料夾名稱與 name 完全一致
frontmatter 包含 name 與 description，description 清楚描述觸發情境
指令層包含 Goals、Steps、Rules、Output Format、Examples 五個要素
用一段「有問題的程式碼」測試，Agent 有依照規定的格式與規則輸出
用一個不相關的問題測試，確認 Skill 沒有被誤觸發
用一段「乾淨的程式碼」測試邊界情況，確認 Agent 不會硬湊問題
