上午課程

生成式 AI
基礎介紹

用工程師最熟悉的思維，10 分鐘搞懂生成式 AI 的本質。

核心觀念

傳統程式 vs 生成式 AI

傳統程式

規則驅動

輸入資料 + 寫死規則（If/Else）= 輸出結果

工程師扮演上帝。所有邏輯、邊界條件全部用程式碼定義好。漏寫一個 if，就會噴 Bug。

生成式 AI

機率驅動

餵入海量資料 + 模型訓練 = 機器自己找出規則

工程師給予「提示與上下文」，AI 基於統計規律去「憑空創造」原本不存在的新內容。

建議時間：5 分鐘
傳統 AI vs 生成式 AI

用途根本不同：分類 vs 創造

傳統 AI

拿來做分類與判斷

  • 判斷這封郵件是不是垃圾信
  • 判斷這張照片是不是貓
  • 預測這筆交易是否詐騙
生成式 AI

拿來創造新東西

  • 幫我寫一封感謝信
  • 畫一隻在太空的貓
  • 把這段 Python 重構成 TypeScript

本質：生成式 AI 是「超級文字接龍機」，根據你給的前文，計算下一個字「最有可能是什麼」。

建議時間：5 分鐘
LLM

大型語言模型（LLM）是什麼？

大腦

壓縮的超大資料庫

讀完了整個網際網路，把知識壓縮進數十億個參數（如 GPT-4、Claude）。

類比

超級實習生

熟讀了 Stack Overflow、GitHub 所有開源專案、無數技術文件，但沒有獨立思考能力，只是在「模仿」。

核心任務

文字接龍

接收未完成的句子，輸出字典裡每個 Token 接在後面的機率分布，預測下一個字。

建議時間：5 分鐘
LLM 內部運作

「文字加工廠」5 道工序

① Tokenize把句子切成 Token 積木，轉成數字編號
② Embed每個 Token ID 轉成高維度向量，意思相近的距離相近
③ Layers向量進入多層神經網路，注意力機制理解上下文
④ LM Head最終向量跟字典所有 Token 計算相似度，打分數
⑤ Softmax分數轉成機率，「擲骰子」選出下一個 Token

Temperature 參數就是控制這顆骰子：調高 → 創意模式，調低 → 保守精準。

建議時間：7 分鐘
Token

Token = 語言模型的「文字積木」

  • AI 看不懂人類文字，先把句子拆成一塊塊 Token 才能運算
  • 積木的切法和我們直覺不一樣：「Apple」、「APPLE」、「 Apple」（前面有空格）是三塊不同 Token
  • 中文有時數個字合成一個 Token，有時一個字被切成多塊
  • 標點符號（！、"）也是獨立的 Token
為什麼計費用 Token？

因為 Token 是 AI 實際運算的單位，不是字數。每個 Token 有唯一的數字編號，再轉成向量進行矩陣運算。

Context Window = AI 的「工作記憶」，超過上限就會開始忘記前面的對話。

建議時間：6 分鐘
Token 的語意特性

積木會互相連結，蘊含語意相似性

例 1：Apple

模型知道「Apple」同時接近「banana（水果）」也接近「MacBook、iPhone（科技公司）」，能依上下文區分。

例 2：文字接龍

輸入「天氣」，模型先在內部想到英文「weather / forecast」，最後才輸出中文「天氣預報」。

例 3：中文姓氏

給模型「王」這個 Token，它知道對應英文「King」，也知道和另一個姓氏「黃」意思接近。

這就是「Embedding」的魔法：意義相近的字在高維度空間中距離相近。

建議時間：5 分鐘
Prompt Engineering

Prompt 是工程師的新型態程式語言

過去

用程式語言下指令

Python / JavaScript 對機器說「做這件事」，邏輯要精確、沒有歧義。

現在

用自然語言下指令

Prompt Engineering 的本質是透過精確的角色設定、背景脈絡、限制條件，收斂 AI 的機率分布，讓它輸出我們要的結果。

Prompt 越模糊 → AI 自由發揮空間越大 → 越容易「幻覺」。Prompt 越精確 → 輸出越可預期。

建議時間：6 分鐘
Prompt Engineering 實戰

好 Prompt 的 5 個欄位

1. 角色你是一個資深前端工程師
2. 背景目前狀況與相關檔案
3. 限制不能改什麼、偏好什麼
4. 輸出要程式、摘要、表格或步驟
5. 驗收如何確認任務完成
模板：
你是一個 [角色]。背景是 [目前狀況]。
請幫我 [目標]，限制是 [限制條件]。
完成後請 [輸出格式]。驗收方式是 [測試或檢查方式]。
建議時間：8 分鐘
AI 開發工具 ①

Claude Code：能操作專案的 AI 協作者

Read

讀懂專案

瀏覽資料夾、閱讀檔案、找出入口點、理解程式結構。

Change

修改內容

編輯程式碼、文件、設定檔，並盡量遵守現有風格。

Verify

驗證結果

執行指令、跑測試、看錯誤訊息，回報修改結果。

在終端機輸入 claude 即可啟動。把它當成會問問題、會動手的 Junior Pair。

建議時間：7 分鐘（含示範）
AI 開發工具 ②

Codex：OpenAI 的雲端 AI 工程師

共同點
  • 都能讀懂整個專案，理解架構與邏輯
  • 都能修改程式碼、跑指令、回報結果
  • 都支援自然語言指令（Prompt）
差異
  • Claude Code：本機終端機，即時互動，支援 Claude 模型
  • Codex：雲端沙箱環境，任務可平行執行，支援 GPT 模型
  • 兩者可互補使用，選擇適合工作情境的工具
建議時間：7 分鐘（含示範）
Agent Mode

AI Agent：LLM ＋ 工具 ＝ 自主數位員工

ReAct 模式（最常見）

思考 → 行動 → 觀察 → 再思考，不斷循環直到任務完成

  • LLM 先思考要做什麼
  • 主程式執行對應工具（讀寫檔、呼叫 API）
  • 把結果當觀察回傳給 LLM，繼續循環
Plan & Execute 模式

先列完整計畫，再逐步執行並動態修正

  • 規劃模型先制定步驟清單
  • 執行 Agent 逐步完成每個步驟
  • 根據結果動態修改後續計畫

工具包括：讀寫檔案、執行終端機指令、呼叫 API、瀏覽網頁。重要操作仍需人工審查確認。

建議時間：8 分鐘
MCP

MCP：AI 工具的「Type-C 通用接孔」

過去的痛點

規格不統一

每個 AI 平台（OpenAI、Anthropic、Google）的工具串接規格不同。開發者寫好一個工具，必須為每個平台各改寫一套程式碼，非常低效。

MCP 的解法

統一工具介接標準

遵守 MCP 規範開發工具，就像打造 Type-C 接頭：只需寫一次，所有支援 MCP 的 Agent 平台都能直接「隨插即用」。

完整架構

LLM（大腦） ＋ Tool 外部工具（四肢） ＋ MCP（萬用卡榫） ＝ AI Agent（自主數位員工）

建議時間：5 分鐘
Agent Skill

Agent Skill：給 Agent 的專屬操作說明書

  • Agent 具備工具能力，但不知道你的私人習慣與格式要求
  • Agent Skill 是一份 Markdown 說明文件，提前寫好讓 Agent 執行特定任務的 SOP
  • 省去每次提問都要重複輸入長串提示詞的麻煩
目標最終要做什麼
步驟先做什麼、再做什麼
規則判斷邏輯
格式輸出的排版要求
MCP vs Agent Skill
  • MCP：擴充 Agent 的「能力」（提供工具，如查天氣、讀檔案）
  • Agent Skill：定義 Agent 的「行為準則」（提供說明書，規定步驟與格式）

命名規則：資料夾名 = Skill 名稱，檔案必須命名為 Skill.md（首字母大寫）

建議時間：5 分鐘

Afternoon Overview

下午三大實作任務

任務 A

Debug 任務

用 AI 快速了解一個陌生專案的架構，找出程式碼裡的技術債與問題點，整理成報告。

任務 B

UI Design 任務

用自然語言描述需求，引導 AI 生成符合現代規範的前端介面（卡路里紀錄 App）。

任務 C

API Test 任務

實作一個調用營養成分 API 的前端應用，讓使用者輸入食物名稱取得營養資訊。

每個任務都有獨立練習環境，點連結另開新視窗，邊看邊做。

實作技巧 ①

用 AI 快速了解一個陌生專案

請先閱讀這個專案，不要修改任何檔案。

請回覆：
1. 專案用途是什麼？
2. 主要資料夾與檔案結構
3. 程式入口點在哪？
4. 本機如何啟動？
5. 你建議我先看哪 3 個檔案？
  • 先要求「不要修改檔案」，把探索和實作分開
  • 適合接手新專案、讀別人的程式、快速找方向
建議時間：10 分鐘
實作技巧 ②

用 AI 找出程式碼裡的問題點

  • 技術債：過期寫法、冗餘程式碼、複雜度過高的函式
  • 安全漏洞：未驗證的輸入、硬編碼的密碼、SQL Injection 風險
  • 命名不一致：camelCase / snake_case 混用、語意不明的變數名
Prompt 範例 
請分析這個專案，
找出以下問題：
- 技術債或過期寫法
- 潛在安全風險
- 命名不一致問題

以條列格式整理，
並標明位於哪個檔案。
建議時間：10 分鐘
實作技巧 ③④

整理問題報告 & 請 AI 提供改善程式碼

Step 1請 AI 列出所有問題點，按嚴重程度排序
Step 2選擇優先處理的問題，要求 AI 提供改善建議
Step 3請 AI 提供修改後的程式碼，說明修改理由
Step 4審查 Diff，確認修改範圍符合預期再接受

記住：AI 提供的程式碼需要你來審查。看 Diff、問理由、驗收測試是你的責任。

建議時間：10 分鐘

目標：在不熟悉新專案的狀態下，利用 AI 快速釐清程式脈絡，找出問題點（過期寫法、命名不一致），並整理成一份改進建議報告。

Best Practices

跟 AI 工具協作的基本節奏

Explore先讀懂專案，不急著改
Plan確認修改範圍與限制
Implement小步修改，保持可回滾
Verify跑測試或手動驗收
Summarize整理成果與潛在風險
推薦工作流：
「先分析，不修改」→「提出方案」→「最小範圍實作」→「跑測試」→「總結 diff」
建議時間：5 分鐘
Common Mistakes

初學者常見錯誤與正確做法

避免
  • 一次丟太大的任務
  • 沒有說清楚限制條件
  • 沒有要求驗證步驟
  • 不看 Diff 就直接接受修改
  • 把 secret / API key 寫進 Prompt
改成
  • 拆成「探索 → 修改 → 驗證」三個步驟
  • 明確列出不能碰的範圍
  • 要求測試指令與手動驗收步驟
  • 看 Diff，請 AI 解釋修改理由
  • 用環境變數或佔位符代替機密資訊
建議時間：5 分鐘
Takeaway Template

課後可直接複製使用的 Prompt

請先閱讀目前專案，不要修改檔案。
請說明專案架構、入口點、啟動方式，以及你建議先看的 3 個檔案。

接著請幫我完成：[具體任務]

限制：
- [限制 1：例如不要改資料庫 schema]
- [限制 2：例如優先沿用現有 component 風格]

完成後請：
1. 說明改了哪些檔案，以及修改理由
2. 跑可用的測試（沒有就說明原因）
3. 提供 3 個手動驗收步驟
4. 列出你認為還有風險的地方
建議時間：3 分鐘
Closing

把 AI 工具用好，不是靠一句神奇指令

  • 清楚描述目標，讓 AI 知道終點在哪
  • 給出限制，讓 AI 知道邊界在哪
  • 要求驗證，讓結果能被檢查
  • 審查 Diff，保留工程師的判斷權
  • AI 是你的 Junior Pair，不是你的替代品

最實用的心法：把大任務拆成「理解 → 修改 → 驗證 → 總結」。
