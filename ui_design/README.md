# 任務2
## 目標： 
使用自然語言或 Markdown 佈局描述，引導 AI 生成符合現代前端規範的介面程式碼。



## 實作步驟規劃

本實作預計耗時 **45 至 60 分鐘**，目標是讓學員僅透過 HTML, CSS (可採用 Tailwind CDN 提高效率) 與原生 JavaScript，產出一個具備動態互動功能的功能頁面。

### 步驟一：拆解 UI 佈局與元件規格（10 分鐘）

在讓 AI 寫 Code 之前，先引導學員進行介面拆解，這是維持生成品質的關鍵。

* **介面核心區塊：**
1. **頂部狀態欄（Header/Summary）：** 顯示今日目標卡路里、已攝取卡路里、剩餘卡路里的進度條（Progress Bar）。
2. **左側/中央搜尋區（Search Section）：** 美食搜尋輸入框、熱門標籤（如：高蛋白、低碳水）、搜尋結果列表（包含食物圖片、名稱、卡路里數值、以及一個「加入紀錄」的按鈕）。
3. **右側/底部紀錄清單（Diary Section）：** 今日已添加的食物清單，支援動態刪除、自動重新計算總熱量。



### 步驟二：使用「結構化 Prompt」生成基礎 HTML 與 Tailwind 佈局（15 分鐘）

引導學員不要直接發送「幫我做一個卡路里 App」這種模糊的指令，而是撰寫明確包含**角色設定、版面結構、樣式規範**的 Prompt。

> **課堂實作範例 Prompt：**
> ```text
> 角色：你是一個資深的前端工程師與 UI 設計師。
> 任務：請使用單一 HTML 檔案（內嵌 Tailwind CSS CDN 與原生 JavaScript）建立一個「美食搜尋與卡路里紀錄」的 Dashboard 介面。
> 
> 版面佈局要求（RWD 行動優先）：
> 1. 頂部大盤：呈現今日卡路里儀表板，包含「目標 2000 kcal」、「已攝取 0 kcal」、「剩餘 2000 kcal」三個數值區塊，並包含一個橫向的視覺進度條。
> 2. 下方雙欄配置（桌面版 md:grid-cols-3）：
>    - 左側欄（佔 2 欄）：美食搜尋區。包含一個 input 欄位與一個搜尋按鈕。下方預留一個用於動態渲染搜尋結果的卡片容器（Grid 佈局）。
>    - 右側欄（佔 1 欄）：今日飲食日記清單。顯示已加入的食物與熱量，底部顯示當前加總。
> 
> 視覺風格：採用乾淨的健康科技風格（背景 slate-50，主要按鈕與進度條使用 emerald-500 綠色系）。
> 
> 輸出限制：請先提供完整的 HTML 與 CSS 佈局結構，JavaScript 邏輯先留白，我們稍後處理。
> 
> ```
>

### 步驟三：引導 JavaScript 互動邏輯生成（20 分鐘）

當 UI 結構產生完畢後，學員需引導 AI 生成控制 DOM 的 JS 程式碼。此處先不串接真實 API，而是請 AI **建立 Mock Data（模擬數據）** 進行前端邏輯測試。

> **課堂實作範例 Prompt：**

```text
> 基於剛才生成的 HTML 結構，請幫我撰寫原生的 JavaScript 程式碼：
> 1. 請建立一個包含 5 筆食物的靜態模擬陣列（Mock Data），欄位包含 id, name, calories, tags（例如：雞胸肉沙拉 350卡、牛肉麵 650卡）。
> 2. 撰寫搜尋邏輯：當使用者在輸入框打字並點擊搜尋時，過濾出符合名稱的食物，並動態渲染（innerHTML）到左側的搜尋結果區。
> 3. 撰寫新增功能：點擊食物卡片上的「加入紀錄」按鈕時，將該食物推進（push）到一個名為 `todayLog` 的陣列中。
> 4. 撰寫更新與渲染邏輯：每次 `todayLog` 變更時，重新計算總熱量、更新頂部儀表板的數值與進度條長度，並動態更新右側的飲食日記清單（清單內的項目需包含一個「刪除」按鈕，點擊可移除該項目）。
> ```

## 講師注意事項與提醒（技術細節與潛在坑點）

在帶領工程師學員進行這段實作時，講師應主動提示以下 3 個實務開發會遇到的不確定性，以展現課程的專業深度：

### 1. 狀態同步（State Management）與 DOM 渲染的混亂
*   **風險：** AI 在生成原生 JS 時，往往會寫出高度耦合的程式碼（例如在「新增按鈕」的 click event 裡面直接去修改 DOM，同時又去改計算數值）。當功能變多時，程式碼會極難維護。
*   **講師提醒：** 引導學員檢視 AI 寫出的結構，**要求 AI 採用「資料驅動（Data-Driven）」的觀念來重構**。即：定義一個全域狀態 `state = { searchResults: [], todayLog: [] }`，所有操作只修改 state，再由一個獨立的 `render()` 函式統一刷新 UI。這能幫工程師建立良好的元件開發思維。

### 2. Tailwind CSS 動態類名（Dynamic Class Names）失效問題
*   **風險：** 工程師常犯的坑。如果學員要求 AI 根據剩餘卡路里的多寡，動態拼接 Tailwind 類名（例如：`class="bg-${color}-500"`），Tailwind 的編譯器或 CDN 是無法辨識這種動態字串的，會導致樣式直接消失。
*   **講師提醒：** 提醒學員在 Prompt 中限定 AI：**「如果遇到動態樣式（如進度條長度或顏色變化），請直接操作 DOM 的 `element.style.width` 或使用完整的類名（如 `bg-red-500`、`bg-emerald-500`），禁止使用字串拼接類名。」**

### 3. 進度條百分比的邊界條件（Edge Cases）
*   **風險：** 當用戶狂吃東西，已攝取卡路里超過目標卡路里（例如吃了 2500 卡，目標 2000 卡）時，進度條百分比會變成 $125\%$。此時如果直接將數值寫入 CSS，進度條會直接「爆框」，甚至導致版面變形；剩餘卡路里也會變成負數。
*   **講師提醒：** 要求學員在邏輯層面加上防禦性程式碼（如 `Math.min(percent, 100)`），並引導 AI 設計當超標時，進度條自動變成警示紅色的邏輯。這能有效訓練學員對 AI 產出進行邊界條件審查的能力。
